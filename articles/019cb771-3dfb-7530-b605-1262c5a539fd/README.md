## MCPは死んでもいいけど今は別に死んでないだろ

https://ejholmes.github.io/2026/02/28/mcp-is-dead-long-live-the-cli.html

CLIの世界、つまりシェルは、自由なバイト列を組み合わせたオプションを与え、入出力として生のバイト列を受け渡す非構造的な空間です。そしてLLMがこの空間で比較的うまく振る舞えるのは、著名なツールについて膨大な学習データを持っているかつ、これから使わせるツールの操作が既知のツールと抽象構造を共有しており推測可能の場合に限ります。そのうえこれはあくまで統計的な推測です。MCPは、入出力をスキーマとして定義することでこの不確実性に構造を与えます。もちろんLLMがスキーマを処理する過程自体は依然統計的ですが、その出力＝サーバーへの入力をクライアント側でも検証することは容易です。CLIのヘルプを充実させるなり丁寧なエラーメッセージを読ませればLLMが従うことが多いのも事実ですが、MCPが存在する以上、特に未知のツールについてヘルプを読ませて（これもまた確率的に）理解させる合理的な理由がありません。

Xで定期的に話題になる「LLMが直接機械語を出力するようになる」という議論がありますが、この議論が広く否定的に受け止められているのは、あまりに緩い制約下での推測と効率に限界があると皆が理解しているからでしょう。「CLIの知識がビルトインされているからMCPは不要」と「機械語の知識がビルトインされているからコンパイラは不要」は、探索空間がどれほど絞られているかの差であって、同じ構造の問題です。もっとも、LLMにできることについての大半の議論は同じ構造になるように思います。ですのでどちらかというと、これをもって「MCPは死んだ」とするのは単に早とちりです（まともに受け取るべきではない煽りタイトルですが）。

さらにMCPは単なるインターフェースではなく、LLMの行動を構造化するフレームワークとしても機能します。先の通りシェルの世界は生のバイト列をやり取りする以上、何の制約もないのだからコンポーザビリティが無限なのは当然です。JSONという構造化された出力に対するjqはもちろん、そうでなくてもgrepやsort、uniqueはたしかに動作します。しかしこれはコンポーザビリティそのものを確率的に取り出しているに過ぎず、そのうえ常に必要な柔軟性でもありません。`terraform show -json | jq ...`で常に決まったフィルタを何度も行うなら、架空の`my-tf-search-mcp - search(keyword)`のほうが確実に動作します。タスクの抽象化こそが、先達の開発者たちが何度も何重にも見出してきた、ソフトウェア開発が工学たりえる一側面だと理解しています。そしてこうして抽象化されたタスクの実行方法は一般に複雑であるべきで、コンテキスト圧迫は非効率というより、タスクに内在する不可避なコストであるといえます。他方、フィルタ条件自体を見出して使い捨てで利用したいなら、LLMはまさにこのような未構造化タスクをうまくこなし、CLIの組み合わせの柔軟性を十分に活かすでしょう。繰り返し実行するうちに、もしかするとシェルスクリプトやPythonスクリプトを自律的に作るかもしれません。わずかな引数で実行するだけならスクリプトのままでもよいかもしれませんし、複雑な使い方を持つならスキーマで縛りたくなるかもしれません。それならMCPの出番です。

私自身としては、氏の批判はCLIの絶対視を除けばもっともだと感じます。MCPについてプロトコルと実装両面の成熟度合いに対する悲嘆も、それを無視するように高まる称揚への呆れも、むしろ共感するところです。氏のように優れたエンジニアなら「CLIが勝つ」と主張するのは自然なことですが、問題は結論を一足飛びに急いだことです。CLIにするかMCPにするか、はたまたデスクトップアプリにするかは、理想的にはコアAPI／ライブラリの妥当なフロントエンド選択の問題にすぎません。この分離をうまく行える類のタスクなら、MCPが不要というのはある程度正しいと思います。その場合にはCLIのオプションも似通ったものになるべきでしょうし、MCPによるスキーマの注入はむしろ不要なコンテキストの圧迫になるはずです。つまり、私の主張は最後の一文がほんの少し違うだけです。Ship a good API, then ship a reasonable frontend for both the API and the agents. The agents will figure it out. 問題はこの分離が上手く行えていない場合で、それはおそらくドメイン分析の不足、設計の不備、そもそもLLMに（あるいは人間に！）向いていないタスクかもしれません。逆に「こういうMCPがあればAIと連携できるのに」と思うなら、それはMCPというより、より良いUIやアーキテクチャのヒント、ヘルプ不備の現れなのかもしれませんね。
